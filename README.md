# NEMR

NetEase Music 지역락 자동 갱신 + 세션 관찰기.

한국/일본에서 네트이즈를 쓰면 며칠에 한 번씩 음악이 회색으로 죽는다. NetEaseMusicWorld+
확장을 켠 채로 웹에 한 번 들어가주면 다시 살아난다. 이걸 대신 해주는 컨테이너다.

## 확장이 실제로 하는 일

소스를 읽은 결과다 (`NetEaseMusicWorld++` v1.0.1, 전체 100줄 남짓). 이미지 빌드 시
`ext/background.js`로 굽히니 직접 확인할 수 있다.

**`X-Real-IP: 211.161.244.70` 헤더를 박는다.** `music.163.com`으로 가는 요청에
declarativeNetRequest 규칙으로 하드코딩된 중국 IP를 넣는다. 네트이즈 API가 이 헤더를 믿고
"중국에서 온 요청"으로 처리해서, 회색 대신 진짜 재생 URL을 준다. 프록시도 VPN도 아니고
상수 하나다. **지역락 우회의 전부가 이거다.**

CDN 호스트명을 `m<숫자>.music.126.net` → `m<숫자>c.music.126.net`으로 바꾸는 코드도 있는데,
`++` v1.0.1에서는 실행되지 않는다 — `content-script.js`의 `injectPageScript()`가 script
엘리먼트를 만들고 `src`와 `onload`까지 달아놓고 **DOM에 append를 안 한다.** MV2 코드
(`inject.js`)를 MV3로 옮기다 망가진 걸로 보인다. `+`(플러스 하나)에서는 이 부분이 멀쩡히
동작하고, 실사용상 문제도 없다고 확인됨.

## 어떻게 동작하는가

네트이즈 서버는 **계정별로 "최근 중국 본토 접속 기록"을 들고 있다.** 기록이 있으면 며칠간
중국에 있는 걸로 쳐주고, 끊기면 외국으로 판정해서 회색 처리한다. 이 기록은 서버 쪽에 있으니
확장 코드에는 안 보인다 — 확장은 헤더만 보내고, 기억은 저쪽이 한다.

그래서 폰 앱이나 데스크톱 클라이언트도 같이 살아난다. 걔들은 확장의 도움을 전혀 못 받고
진짜 한국 IP로 나가는데도 되는 이유가 이거다. 웹에서 헤더 달린 요청 한 번 = 계정 전체가
며칠 연장.

**즉 자동화해야 할 건 딱 하나다: 헤더 달린 요청을 주기적으로 한 번씩 보내기.**
이 컨테이너가 6시간마다 그걸 한다.

## 자동화되지 않는 것: 로그인

로그인은 별개 문제고, **자동화 못 한다.**

매일 웹에 들어가서 새로고침을 해줘도 어느 순간 로그인이 풀려 있다. 즉 세션이 노는 동안
만료되는 게 아니라 **활동과 무관하게 강제로 끊긴다.** 계정을 계속 두들겨봐야 소용없다는 뜻이고,
이 컨테이너도 그건 못 막는다. 2차 인증은 애초에 이런 걸 막으라고 있는 거다.

이유는 아직 모른다. 지금 계정은 웹(중국 헤더) + 폰(한국) + 데스크톱(한국)으로 동시에 보이는데,
리스크 컨트롤이 그 모순을 읽고 끊는 건지, 그냥 해외 계정 세션 TTL이 짧은 건지 구분이 안 된다.
`data/probe.jsonl`에 `loggedIn`이 언제 떨어지는지 쌓이면 그때 다시 본다.

그래서 이 도구의 현실적 효용은 **"매일 하던 일을 안 하게 되고, 몇 주에 한 번 QR만 스캔"** 이다.

## Portainer로 띄우기

Stacks → Add stack → Repository:

- Repository URL: `https://github.com/NR2BJ/NEMR`
- Compose path: `docker-compose.yml`

그대로 띄우면 본업(6시간마다 캐시 갱신)은 바로 돈다. `TRACK_IDS`는 안 넣어도 되고,
넣으면 아래 관찰 기록이 같이 쌓인다. 회색으로 변하는 곡 ID 몇 개를 쉼표로 구분해서 넣는다.
트랙 URL에서 딴다: `https://music.163.com/#/song?id=1234567` → `1234567`

원래부터 재생 가능한 곡을 넣으면 관찰값이 무의미해지니, **확실히 회색 되는 곡**으로 고를 것.

| 환경변수 | 기본값 | 설명 |
|---|---|---|
| `TRACK_IDS` | (없음) | 관찰할 곡 ID, 쉼표 구분. 비워도 갱신은 동작 |
| `PROBE_INTERVAL_SEC` | `21600` (6h) | 갱신 주기 |
| `SETTLE_MS` | `10000` | 페이지 로드 후 확장이 붙기까지 대기 |

## 최초 로그인 (한 번만)

**이건 Portainer UI로는 안 되고 서버 셸이 필요하다.** 스택을 띄우기 전에 먼저 해야 한다 —
로그인이 없으면 갱신할 세션 자체가 없다.

서버에 화면이 없으니 VNC로 붙어서 QR을 스캔한다. 리포지토리를 클론한 곳에서:

```bash
docker compose run --rm --service-ports login
```

다른 터미널에서 터널을 뚫고:

```bash
ssh -L 5900:127.0.0.1:5900 <user>@<server>
```

`vnc://127.0.0.1:5900` 접속 → 로그인 → 브라우저는 10분간 열려 있다가 알아서 닫힌다.
프로필은 `./data/profile`에 남아 이후 실행에 재사용된다.

VNC는 컨테이너 안에서만 열리고 호스트 루프백에만 노출되니, SSH 터널 없이는 접근할 수 없다.
로그인이 풀리면 이 과정을 다시 해야 한다.

## 매 실행마다 기록되는 것

같은 곡을 **두 경로로** 조회해서 `data/probe.jsonl`에 한 줄 남긴다.

- `clientCold` — 프로필 쿠키만 빌려서 node에서 직접. 한국 IP, 헤더 없음.
  **폰 앱이 보는 세상이다.**
- `browser` — 확장 켜진 크롬으로. 헤더 붙음. 항상 언락돼야 정상.
- `clientWarm` — cold와 완전히 동일한 요청을, browser 직후에 다시.

순서가 중요하다. 브라우저 방문이 곧 캐시를 갱신하는 행위라, cold를 먼저 재지 않으면
probe가 자기 관찰을 오염시킨다.

```bash
jq -c '{at, ext: .extension.active, login: .account.loggedIn,
        cold: .clientCold.playable, browser: .browser.playable, warm: .clientWarm.playable}' \
   data/probe.jsonl
```

| 관찰 | 의미 |
|---|---|
| `ext: false` | 확장이 안 붙었다. 나머지 값은 볼 필요 없음 |
| `cold: 0`, `warm: N` | 캐시가 만료돼 있었고, 이번 실행이 되살렸다. 정상 동작 |
| `cold: N`, `warm: N` | 아직 안 만료됨. 주기가 TTL보다 짧다는 뜻 — 정상 |
| `login: false` | QR 스캔하러 갈 시간 |

`data/shots/`에 매 실행 스크린샷이 남는다.

## TTL을 재고 싶다면

6시간 주기로 돌리면 캐시가 항상 따뜻해서 **TTL은 영영 못 잰다.** 관찰 도구와 해결책이
같은 물건이라 생기는 트레이드오프다. 굳이 재고 싶으면 `PROBE_INTERVAL_SEC`를 크게 잡고
(예: 6h → 12h → 24h) `cold`가 0으로 떨어지는 지점을 찾으면 된다. 그동안 웹 플레이어를
손으로 새로고침하면 측정이 망가지니 주의. 폰/데스크톱은 써도 된다 — 헤더를 안 보내니까.

실용적으로는 알 필요 없다. 6시간은 "며칠"보다 한참 짧으니 안전하다.

## 나중에 더 줄일 여지

지역락 우회가 헤더 하나뿐이므로, 로그인만 살아 있으면 크롬도 확장도 필요 없다.
쿠키 들고 `curl -H 'X-Real-IP: 211.161.244.70'` 한 줄이면 끝일 수 있다. 다만 네트이즈가
브라우저 지문이나 다른 신호를 같이 보는지 확인이 안 돼서, 일단은 진짜 크롬으로 간다.
`clientCold`/`clientWarm`이 그 답을 조금씩 모아줄 것이다.

## 알려진 미확인 사항

- 확장 분석은 `++` v1.0.1 기준. 실사용 중인 `+`는 CDN 재작성이 살아있다는 점이 다르다.
- `/api/nuser/account/get`, `/api/song/enhance/player/url`은 비공식 API다. 언제든 바뀐다.
- `clientCold`는 데스크톱 클라이언트 UA를 흉내내지만 진짜 클라이언트는 아니다.
  네트이즈가 UA 말고 다른 걸로 식별한다면 이 근사는 깨진다.
- 하드코딩된 `211.161.244.70`이 언제까지 유효할지 모른다. 확장 마지막 업데이트는 2025년 5월.
- Docker 빌드는 검증하지 못했다 (작성 환경에 Docker가 없었음). 확장 다운로드/언팩만 실제 확인.
